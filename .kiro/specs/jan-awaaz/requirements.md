# Requirements Document: Jan-Awaaz

## Introduction

Jan-Awaaz is a voice-first Progressive Web Application (PWA) designed to help rural citizens in India access government schemes through voice interaction in major Indian regional languages. The system addresses critical barriers including language limitations, complex documentation requirements, lack of scheme awareness, and low digital literacy. The platform enables users to describe their situation verbally, verify documents through image capture, and receive personalized guidance on eligible schemes with clear directions to the nearest Common Service Centre (CSC).

**Current Implementation Status:** The application is deployed on Vercel as a Next.js PWA with AWS backend services. Users interact through Web Speech API (Chrome/Edge browsers), capture documents via mobile camera, and receive AI-generated referral cards with CSC center locations and Google Maps integration.

## Glossary

- **System**: The Jan-Awaaz Progressive Web Application (PWA) and AWS backend services
- **User**: Rural citizen seeking government scheme assistance (primary user)
- **CSC_Operator**: Common Service Centre operator who assists with applications (secondary user)
- **Voice_Input**: Audio captured from user's microphone using Web Speech API in major Indian regional languages (Hindi, Tamil, Telugu, Kannada, Malayalam, Marathi, Bengali, Gujarati, Punjabi, Odia)
- **Voice_Recognition**: Web Speech API (browser-based) that converts voice to text (Chrome/Edge only)
- **Voice_Output**: Audio response generated by Web Speech API in local language
- **Document_Image**: Photo of physical document captured via mobile camera
- **Verification_Service**: AWS Textract service that extracts and validates document data
- **Scheme_Matcher**: Amazon Bedrock Knowledge Base with Claude 3 Haiku that matches user situations to schemes
- **Referral_Card**: Digital summary card (Parchee) showing scheme eligibility, requirements, and CSC center location
- **CSC_Locator**: AI-powered service using Claude 3 Haiku that identifies nearest Common Service Centre
- **Session**: User interaction tracked by mobile phone number in DynamoDB
- **PWA**: Progressive Web Application - installable web app that works offline

## Requirements

### Requirement 1: Voice-Based User Input

**User Story:** As a rural citizen with low digital literacy, I want to describe my situation using voice in my regional language, so that I can access government schemes without typing or reading English.

**Implementation Status:** ✅ IMPLEMENTED using Web Speech API (browser-based)

#### Acceptance Criteria

1. WHEN a User speaks into the microphone, THE System SHALL capture the Voice_Input in any supported Indian regional language using Web Speech API
2. WHEN Voice_Input is captured, THE System SHALL convert it to text in real-time using browser-based speech recognition
3. WHEN the User speaks for more than 60 seconds continuously, THE System SHALL process the audio continuously without losing context
4. IF the Voice_Recognition fails to process audio, THEN THE System SHALL provide a text input fallback for manual entry
5. WHEN Voice_Input contains background noise, THE Web Speech API SHALL filter noise and extract the primary voice signal

**Implementation Notes:**
- Uses Web Speech API (works in Chrome/Edge browsers only)
- Supports all 10 Indian languages with language-specific recognition models
- Provides manual text input as fallback for unsupported browsers
- No interim results to prevent text repetition on mobile devices

### Requirement 2: Scheme Matching from User Story

**User Story:** As a rural citizen, I want the system to understand my life situation and match me to relevant government schemes, so that I don't need to know scheme names or eligibility criteria.

**Implementation Status:** ✅ IMPLEMENTED using Amazon Bedrock Knowledge Base with Claude 3 Haiku

#### Acceptance Criteria

1. WHEN transcribed text is received, THE Scheme_Matcher SHALL query the Bedrock Knowledge Base with the user's situation using Claude 3 Haiku
2. WHEN querying the Knowledge Base, THE Scheme_Matcher SHALL return only schemes that exist in the provided PDF documents (no hallucination)
3. WHEN multiple schemes match the user's situation, THE System SHALL rank them by relevance and present the top match
4. WHEN no schemes match the user's situation, THE System SHALL respond with Voice_Output explaining no matches were found and ask clarifying questions
5. WHEN a scheme match is found, THE System SHALL extract scheme name, document count (number), and document names (comma-separated list) in parallel API calls

**Implementation Notes:**
- Uses parallel API calls: one for conversational response (in user's language), one for metadata extraction (English)
- Metadata includes: scheme name, document count as number, document names as comma-separated list
- Takes only first 3 documents as mandatory from scheme requirements
- Haiku speaks to user in their selected language while metadata is extracted in English for parsing
- Query expansion maps colloquial phrases to scheme keywords (e.g., "husband died" → "widow pension")

### Requirement 3: Document Verification via Image Capture

**User Story:** As a rural citizen, I want to verify my documents before traveling to a government office, so that I don't waste time and money on trips that will be rejected.

**Implementation Status:** ✅ IMPLEMENTED with camera capture and skip functionality

#### Acceptance Criteria

1. WHEN a User captures a Document_Image via camera, THE System SHALL upload it to AWS S3 for storage
2. WHEN the Verification_Service processes a Document_Image, THE System SHALL extract text using AWS Textract
3. WHEN an Aadhaar card is uploaded, THE Verification_Service SHALL validate the 12-digit number format
4. WHEN an Income Certificate is uploaded, THE Verification_Service SHALL check for required government seal and issuing authority signature
5. WHEN a document is expired based on extracted date, THE System SHALL notify the User via Voice_Output with the expiration date
6. WHEN a document is missing a required seal or signature, THE System SHALL notify the User via Voice_Output specifying what is missing
7. WHEN a Document_Image is too blurry or dark, THE System SHALL prompt the User to recapture with better lighting
8. WHEN a User does not have a document, THE System SHALL allow them to skip it and show it later at CSC center

**Implementation Notes:**
- Skip button available before camera access and during capture for all documents
- Camera stops after skip and doesn't auto-start, giving user choice for each document
- Document detection works for all 10 Indian languages
- Users can skip documents they don't have - they can provide them at CSC center
- System tracks documents uploaded vs documents required

### Requirement 4: Voice-Based Feedback and Guidance

**User Story:** As a rural citizen who cannot read, I want to receive all information through voice in my local language, so that I can understand the guidance without assistance.

**Implementation Status:** ✅ IMPLEMENTED using Web Speech API for text-to-speech

#### Acceptance Criteria

1. WHEN the System generates a response, THE Voice_Output SHALL be produced in the same language as the Voice_Input using Web Speech API
2. WHEN scheme matches are found, THE System SHALL use Voice_Output to describe each scheme in simple terms in the user's language
3. WHEN document verification completes, THE System SHALL use Voice_Output to confirm validity or explain issues
4. WHEN Voice_Output is generated, THE System SHALL use browser-based text-to-speech to synthesize speech
5. WHEN the User requests information to be repeated, THE System SHALL replay the last Voice_Output
6. WHEN Voice_Output is playing, THE System SHALL stop voice recognition to prevent echo/feedback loop

**Implementation Notes:**
- Uses Web Speech API for text-to-speech (browser-based, works in Chrome/Edge)
- Stops recording before speaking to prevent microphone from picking up speaker output
- Prevents echo/feedback loop that occurred when using headphones
- Voice output in user's selected language with appropriate voice selection

### Requirement 5: Location-Based CSC Center Routing

**User Story:** As a rural citizen, I want to know exactly which CSC center to visit and how to get there, so that I go to the correct location on my first attempt.

**Implementation Status:** ✅ IMPLEMENTED using AI-powered CSC search with Google Maps integration

#### Acceptance Criteria

1. WHEN a scheme requires application submission, THE CSC_Locator SHALL identify the nearest Common Service Centre using AI search
2. WHEN the User provides their city/area name, THE System SHALL search for CSC centers in that location using Claude 3 Haiku
3. WHEN a CSC center is found, THE System SHALL extract name, address, landmark, and contact number
4. WHEN CSC information is displayed, THE System SHALL translate it to the user's selected language
5. WHEN the User views the referral card, THE System SHALL provide a Google Maps button to open directions
6. WHEN the Google Maps button is clicked, THE System SHALL use Google Maps Geocoding API to convert address to exact coordinates
7. IF geocoding succeeds, THEN THE System SHALL open Google Maps with precise lat/lng coordinates
8. IF geocoding fails, THEN THE System SHALL fall back to address search mode in Google Maps

**Implementation Notes:**
- Uses Claude 3 Haiku to search for CSC centers based on user's location
- Extracts clean address information (removes English labels like "Name:", "Address:")
- Translates CSC information to user's language for display
- Google Maps integration with Geocoding API for precise location
- Fallback to search mode if API key not provided or geocoding fails
- "Open in Maps" button with 📍 emoji, translated in all 10 languages

### Requirement 6: Digital Referral Card Generation

**User Story:** As a rural citizen, I want a simple visual card that I can show at the CSC center, so that the operator understands what I need without me explaining everything again.

**Implementation Status:** ✅ IMPLEMENTED with SVG generation and download functionality

#### Acceptance Criteria

1. WHEN a User is matched to a scheme, THE System SHALL generate a Referral_Card containing scheme name, user phone number, required documents, and CSC center location
2. WHEN the Referral_Card is displayed, THE System SHALL use heavy iconography alongside text to ensure comprehension by illiterate users
3. WHEN the Referral_Card is generated, THE System SHALL display it on screen with large, clear text and icons
4. WHEN the Referral_Card is displayed, THE System SHALL allow the User to download it as an SVG image
5. WHEN a CSC_Operator views the Referral_Card, THE System SHALL display all eligibility criteria and document checklist
6. WHEN the Referral_Card is generated, THE System SHALL include a timestamp and unique reference number
7. WHEN the User clicks download, THE System SHALL use a proxy API to bypass CORS restrictions and download the card
8. WHEN the Referral_Card is displayed, THE System SHALL show CSC center information with Google Maps integration

**Implementation Notes:**
- Referral cards generated as SVG and stored in S3 bucket
- Download proxy API (`/api/download-card`) bypasses CORS restrictions
- Downloads as `referral-card-[reference-number].svg`
- Black border added to card for better visibility
- Google Maps button with geocoding for precise location
- Share button for sharing card via mobile share API
- All text translated to user's selected language

### Requirement 7: Phone Number-Based Session Management

**User Story:** As a rural citizen without email or literacy skills, I want to use my phone number to access my information, so that I can return to my application without complex login processes.

#### Acceptance Criteria

1. WHEN a User first accesses the System, THE System SHALL request the mobile phone number via Voice_Output
2. WHEN a phone number is provided via Voice_Input, THE System SHALL create or retrieve a Session associated with that number
3. WHEN a User returns to the System, THE System SHALL recognize the phone number and restore the previous Session
4. WHEN a Session is restored and the status is PENDING_ACTION, THE System SHALL immediately prompt the User via Voice_Output with a contextual question without waiting for user input
5. WHEN a Session is created, THE System SHALL store scheme matches, document verification results, Referral_Cards, and action status
6. WHEN a Session is inactive for 30 days, THE System SHALL archive the data but retain the phone number association

### Requirement 8: Low Bandwidth Optimization

**User Story:** As a rural citizen with limited mobile data and slow internet, I want the app to work efficiently on my connection, so that I can complete my application without excessive data costs or timeouts.

#### Acceptance Criteria

1. WHEN uploading a Document_Image, THE System SHALL compress the image to under 500KB while maintaining readability
2. WHEN generating Voice_Output, THE System SHALL use compressed audio formats optimized for mobile networks
3. WHEN network connectivity is poor, THE System SHALL queue operations and retry automatically
4. WHEN the User is offline, THE System SHALL display a clear message via Voice_Output explaining the connectivity issue
5. WHEN data transfer is in progress, THE System SHALL display a visual progress indicator

### Requirement 9: Multi-Language Support

**User Story:** As a rural citizen who speaks a regional Indian language, I want to interact with the system in my native language, so that I can understand and be understood without language barriers.

#### Acceptance Criteria

1. WHEN the System starts, THE System SHALL detect the User's preferred language from device settings or prompt for selection
2. WHEN the User selects a supported language, THE Transcription_Service SHALL process Voice_Input in that language and THE Voice_Output SHALL be in that language
3. THE System SHALL support Hindi, Tamil, Telugu, Kannada, Malayalam, Marathi, Bengali, Gujarati, Punjabi, and Odia
4. WHEN the Referral_Card is generated, THE System SHALL display text in the User's selected language
5. WHEN the User switches language mid-session, THE System SHALL update all Voice_Output and text displays to the new language

### Requirement 10: Document Type Recognition

**User Story:** As a rural citizen who doesn't know technical document names, I want the system to automatically identify what document I'm showing, so that I don't need to select from confusing menus.

#### Acceptance Criteria

1. WHEN a Document_Image is uploaded, THE Verification_Service SHALL automatically classify it as Aadhaar, Income Certificate, Land Patta, or Other
2. WHEN the document type is identified, THE System SHALL apply type-specific validation rules
3. WHEN an Aadhaar card is identified, THE Verification_Service SHALL extract the 12-digit number, name, and date of birth
4. WHEN an Income Certificate is identified, THE Verification_Service SHALL extract annual income amount, issuing authority, and validity date
5. WHEN a Land Patta is identified, THE Verification_Service SHALL extract survey number, area, and owner name
6. IF the document type cannot be determined, THEN THE System SHALL prompt the User via Voice_Output to describe what document they are showing

### Requirement 11: Error Handling and Recovery

**User Story:** As a rural citizen with limited technical knowledge, I want clear guidance when something goes wrong, so that I can fix issues without getting stuck or frustrated.

#### Acceptance Criteria

1. IF the Transcription_Service fails to understand Voice_Input, THEN THE System SHALL ask the User to repeat more slowly via Voice_Output
2. IF the Verification_Service cannot process a Document_Image, THEN THE System SHALL explain the issue via Voice_Output and suggest recapture
3. IF the Scheme_Matcher returns no results, THEN THE System SHALL ask clarifying questions via Voice_Output to gather more information
4. IF network connectivity is lost during an operation, THEN THE System SHALL save progress and resume when connectivity returns
5. WHEN any service error occurs, THE System SHALL log the error with Session identifier for debugging without exposing technical details to the User

### Requirement 12: Scheme Information Accuracy

**User Story:** As a rural citizen relying on this system for critical benefits, I want accurate and up-to-date scheme information, so that I don't pursue schemes I'm not eligible for or miss ones I qualify for.

#### Acceptance Criteria

1. WHEN the Scheme_Matcher queries the Knowledge Base, THE System SHALL only return schemes that exist in the verified PDF documents
2. WHEN scheme eligibility criteria are presented, THE System SHALL display the exact criteria from the source document without modification
3. WHEN a scheme has been updated in the Knowledge Base, THE System SHALL reflect the changes in all new queries
4. THE System SHALL NOT generate or hallucinate scheme names, eligibility criteria, or application processes
5. WHEN a User asks about a scheme not in the Knowledge Base, THE System SHALL clearly state via Voice_Output that the scheme information is not available

### Requirement 13: Performance and Latency Requirements

**User Story:** As a rural citizen with limited patience for technology, I want quick responses to my voice commands, so that the interaction feels natural and I don't give up.

#### Acceptance Criteria

1. WHEN Voice_Input is captured, THE Transcription_Service SHALL return text within 5 seconds
2. WHEN the Scheme_Matcher processes a query, THE System SHALL return results within 5 seconds
3. WHEN a Document_Image is uploaded, THE Verification_Service SHALL complete analysis within 5 seconds
4. WHEN Voice_Output is generated, THE System SHALL begin playback within 2 seconds
5. WHEN the Office_Locator calculates routes, THE System SHALL display the map within 3 seconds

### Requirement 14: Privacy and Data Security

**User Story:** As a rural citizen sharing sensitive personal documents, I want my information to be secure, so that my identity and documents are not misused.

**Implementation Status:** ✅ IMPLEMENTED with AWS encryption and security best practices

#### Acceptance Criteria

1. WHEN a Document_Image is uploaded, THE System SHALL encrypt it during transmission (HTTPS) and storage (S3 server-side encryption)
2. WHEN a Session is created with a phone number, THE System SHALL store it securely in DynamoDB
3. WHEN Voice_Input is processed, THE System SHALL not store raw audio beyond the active Session
4. WHEN a User's data is archived after 30 days, THE System SHALL remove all Document_Images and retain only scheme match history
5. THE System SHALL NOT share User data with third parties without explicit consent

**Implementation Notes:**
- All API calls use HTTPS (TLS encryption)
- S3 buckets use server-side encryption
- DynamoDB sessions encrypted at rest
- No raw audio storage - only transcribed text
- Vercel deployment with environment variables for AWS credentials

### Requirement 15: UI/UX Enhancements for Accessibility

**User Story:** As a rural citizen with low digital literacy, I want a visually clear and easy-to-use interface, so that I can navigate the app without confusion.

**Implementation Status:** ✅ IMPLEMENTED with glassmorphism, borders, and visual improvements

#### Acceptance Criteria

1. WHEN the User views any UI component, THE System SHALL display black borders (4px) around all content boxes for clear visual separation
2. WHEN the User views the main heading, THE System SHALL display it with a glassmorphism effect (frosted glass) for better visibility
3. WHEN the User views language options, THE System SHALL display native language names in darker colors (text-gray-800) for better readability
4. WHEN the User views the background, THE System SHALL display floating multilingual "scheme" words with low opacity and blur effect
5. WHEN the User hovers over floating words, THE System SHALL make them interactive (scale up, change color, pointer cursor)
6. WHEN the User views transcription text, THE System SHALL display it in black color with medium font weight for better visibility
7. WHEN the User views the app on mobile, THE System SHALL stack input and submit button vertically for better layout
8. WHEN the User installs the app, THE System SHALL use logo.png as the favicon and app icon

**Implementation Notes:**
- Black borders (border-4 border-black) on all main UI boxes
- Glassmorphism effect on header (backdrop-blur-md, bg-white/50)
- 33 floating "scheme" words in 10 Indian languages with animations
- Interactive hover effects on floating words (scale, color change)
- Clean gradient background (blue-50 to white) replacing background image
- Transcription text in black (text-black font-medium) instead of gray
- Responsive layout with flex-col sm:flex-row for mobile/desktop
- Logo.png used for favicon, apple-icon, and manifest icons

### Requirement 16: Progressive Web App (PWA) Features

**User Story:** As a rural citizen with limited internet connectivity, I want to install the app on my phone and use it offline when possible, so that I can access it anytime.

**Implementation Status:** ✅ IMPLEMENTED as Next.js PWA with offline support

#### Acceptance Criteria

1. WHEN the User visits the app on mobile, THE System SHALL prompt them to install it as a PWA
2. WHEN the User installs the app, THE System SHALL add an icon to their home screen
3. WHEN the User opens the installed app, THE System SHALL display it in fullscreen mode without browser chrome
4. WHEN the User is offline, THE System SHALL display cached content where possible
5. WHEN the User returns online, THE System SHALL sync any pending operations

**Implementation Notes:**
- Next.js PWA with manifest.json and service worker
- Installable on mobile devices (Android/iOS)
- Logo.png used as app icon
- Offline support via service worker
- Deployed on Vercel with automatic HTTPS
